# JARVIS NEXT v1

## Цель v1

Собрать первую версию, которая уже ощущается продуктом, а не прототипом.

`v1` не пытается уметь всё. Он строит прочное ядро.

## Основа v1

### 1. Один главный shell

На экране есть:

- верхняя зона состояния;
- центральный диалог;
- нижний крупный composer;
- контекстный drawer/overlay для инструментов;
- один режим compact для узких ширин.

Нет:

- постоянной свалки панелей;
- трёх дублирующих навигаций;
- случайных служебных карточек внутри главного потока.

### 2. Первый вход

Первый запуск должен быть частью мира продукта, а не скучной админкой.

Он состоит из:

- пролога;
- активации/профиля;
- базовых модулей;
- входа в рабочий хаб.

### 3. Главные режимы v1

- `Create`
- `Code`
- `Roblox`
- `System`
- `Recreate`

`Research` может войти частично, но не должен раздувать UI.

### 4. Выполнение действий

Каждое реальное действие проходит цепочку:

`понял -> показал план -> dry-run -> подтвердил при необходимости -> выполнил -> показал результат`

### 5. Asset-first слой

Для задач по `Roblox` и визуальному производству JARVIS NEXT сначала ищет:

- готовые деревья;
- prop packs;
- environment packs;
- UI kits;
- creator store assets;
- reusable systems.

Только потом он генерирует кастомную структуру.

## Экраны v1

### Главный экран

Содержит:

- название текущего режима;
- короткий статус;
- живой voice meter;
- большую область диалога;
- нижний composer;
- кнопку открытия drawer.

### Drawer / tool layer

Открывается по контексту и содержит:

- режимы;
- историю;
- project context;
- быстрые системные действия;
- pinned tools.

Он не живёт как постоянная тяжёлая колонка.

### Центр настроек

Отдельная спокойная поверхность.

Внутри:

- `Базовые`
- `Продвинутые`
- `Голос`
- `Проекты`
- `Система`

У каждого важного поля есть `?`.

### Action preview

Показывает:

- что именно будет сделано;
- на чём это сработает;
- почему это считается опасным или нет;
- какую кнопку жмёт пользователь дальше.

### История

Фильтры:

- `чат`
- `системные действия`
- `ошибки`
- `Roblox`
- `код`

### Project context

Минимум для `v1`:

- текущий проект;
- активный режим;
- связанные файлы;
- последние действия;
- pinned context.

## Архитектура v1

### Core

Оркестратор задач, памяти, маршрутизации, режимов и выполнения.

### UI shell

Один front-end shell, который не знает деталей доменных режимов.

### Skills

Навыки подключаются как capability-слой.

### Executors

Отдельный слой выполнения:

- код;
- файлы;
- system actions;
- project actions;
- asset lookup.

### Memory

Держит:

- профиль пользователя;
- проектный контекст;
- историю действий;
- pinned preferences.

## Roblox v1

JARVIS NEXT в режиме `Roblox` обязан уметь:

1. Писать `Luau`.
2. Разбирать `UI`, `gameplay loop`, `doors`, `hub`, `physics`.
3. Предлагать готовые ассеты раньше, чем сочинять новые.
4. Давать связный build plan для фичи.
5. Работать как dev-assistant, а не как теоретик.

## Что не входит в v1

- Telegram как основной контур.
- Домашняя автоматика как центральный сценарий.
- Десятки декоративных режимов.
- Сервисная диагностика как половина интерфейса.
- Дублирующие окна и legacy-мастера.

## Стек

Базовый выбор на новый цикл:

- `Tauri`
- `React`
- `TypeScript`
- `Python` как runtime/AI/execution layer

Причина:

- сильный UI-контур;
- хороший desktop shell;
- можно сохранить Python как рабочее исполнительное ядро.

## Критерий успеха v1

Если пользователь открывает продукт и за 30 секунд понимает:

- где он;
- что можно сделать;
- как начать;
- почему это ощущается цельно,

то `v1` уже удался.
